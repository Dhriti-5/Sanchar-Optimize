# Phase 3 Scalability Configuration

## Overview
Phase 3 introduces distributed data storage with Amazon Timestream and DynamoDB, enabling the system to scale to 10,000+ concurrent users while maintaining <500ms response times.

## Architecture Components

### 1. Amazon Timestream (Time-Series Telemetry)
**Purpose:** Historical network telemetry storage for pattern analysis and shadow zone detection

**Configuration:**
- **Memory Store Retention:** 24 hours (hot data for real-time analysis)
- **Magnetic Store Retention:** 30 days (historical patterns)
- **Write Throughput:** Auto-scales to handle burst writes
- **Query Performance:** Optimized for time-range and location-based queries

**Scaling Characteristics:**
- Handles millions of writes per second
- Pay-per-use pricing (no pre-provisioning)
- Automatic data tiering (memory → magnetic → archival)

### 2. Amazon DynamoDB (Session Management)
**Purpose:** User session tracking, contextual memory, and modality transition history

**Configuration:**
- **Billing Mode:** PAY_PER_REQUEST (on-demand auto-scaling)
- **TTL:** Enabled (automatic cleanup after 7/30 days)
- **Global Secondary Index:** session_id-transition_time-index for fast queries

**Scaling Characteristics:**
- Scales from 0 to 10,000+ concurrent users automatically
- Single-digit millisecond latency at any scale
- No capacity planning required with on-demand mode

### 3. FastAPI Backend
**Current:** Single instance (development)
**Production:** Multiple instances behind AWS Application Load Balancer

**Recommended Deployment:**
```
AWS Lambda + API Gateway (serverless)
or
ECS Fargate (containerized, auto-scaling)
```

**Auto-Scaling Rules:**
- CPU > 70% → Scale out
- Request rate > 1000/min → Scale out
- Response time > 500ms → Scale out

### 4. Redis Cache (Optional)
**Purpose:** Hot cache for frequently accessed data

**Configuration:**
- **Mode:** ElastiCache Redis (AWS managed)
- **Eviction Policy:** LRU (Least Recently Used)
- **TTL:** 5 minutes for prediction results, 1 hour for summaries

## Deployment Modes

### Development Mode
```yaml
Backend: Single FastAPI instance
Timestream: Not required (graceful degradation)
DynamoDB: Not required (graceful degradation)
Redis: Local instance or disabled
```

### Production Mode (AWS)
```yaml
Backend: ECS Fargate (2-10 tasks, auto-scaling)
Load Balancer: Application Load Balancer
Timestream: Fully enabled
DynamoDB: On-demand mode

```

### Edge Mode (Future - Phase 4)

## Cost Optimization

### Timestream
- Use 24-hour memory retention (most queries are recent)
- Set magnetic store to 30 days (sufficient for pattern analysis)
- Enable data compression (automatic)

**Estimated Cost (10,000 users, 1 telemetry/sec):**
- Writes: ~$25/day (864M writes)
- Storage: ~$5/day (5GB magnetic)
- Queries: ~$10/day (1000 queries)
- **Total: ~$40/day (~$1,200/month)**

### DynamoDB
- Use on-demand mode (no wasted capacity)
- Enable TTL for automatic cleanup
- Minimize item size (<1KB per session)

**Estimated Cost (10,000 users, 7-day sessions):**
- Writes: ~$5/day (200K session updates)
- Reads: ~$2/day (100K position checks)
- Storage: ~$1/month (70K sessions × 1KB)
- **Total: ~$7/day (~$210/month)**

### Lambda (Future)
- Use ARM64 (Graviton2) for 20% cost savings
- Set memory to 512MB (optimal for ML inference)
- Use provisioned concurrency only for critical paths

## Security

### Network
- API behind WAF (rate limiting, geo-blocking)
- HTTPS only (TLS 1.3)
- CORS restricted to extension origins

### Data Privacy
- GPS coordinates anonymized to 0.1-degree grid (~11km)
- Session IDs are UUIDs (not personally identifiable)
- No video content stored (only metadata)
- TTL ensures automatic data deletion
